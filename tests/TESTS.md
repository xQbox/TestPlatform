# D09T06 — тесты по методике классов эквивалентности

Формат как в D07T04/D08T05: `testN.txt` подаётся в `stdin`, вывод сравнивается
с `testN.out`. Файлы без завершающего перевода строки и без хвостовых пробелов.

| Директория | Квест | Тестов |
| ------ | ------ | ------ |
| `data_module_entry/` | Quest 1. Modules (нормализация) | 15 |
| `yet_another_decision_module_entry/` | Quest 2. Modules II (YES/NO) | 12 |

## Что НЕ покрыто и почему

- **Quest 3 (main_executable_module)** — в задании не зафиксирован формат вывода,
  эталон был бы выдумкой.
- **Quest 6–12 (s21_string)** — тестовая программа сама печатает SUCCESS/FAIL и не
  читает `stdin`; под модель «`stdin` → сравнение с эталоном» не ложится.
- **Quest 13 (text_processor)** — запускается только как `text_processor -w`, а
  `run_tests.sh` не передаёт аргументы командной строки.

## Допущения (проверить перед использованием!)

Рабочего решения D09T06 нет, эталоны выведены из текста задания. Три места, где
задание допускает разночтения:

1. **Quest 1 — это min-max нормализация**: `(x - min) / (max - min)`. Пример из
   README (`1 2 3 4 5` → `0.00 0.25 0.50 0.75 1.00`) сам по себе неоднозначен —
   ранговая нормализация даёт на нём тот же ответ. Различает их только `test5`
   (`1 2 4` → `0.00 0.33 1.00`; ранговая дала бы `0.00 0.50 1.00`). Если модуль
   реализует другую формулу — упадёт именно `test5`.
2. **Quest 2 — среднее считается по исходным, а не по нормализованным данным.**
   Здесь пример из README однозначен: для `1 2 3 4` среднее исходных = 2.5 ≥ 0.618
   → YES, а среднее нормализованных = 0.5 < 0.618 дало бы NO.
3. **Quest 1 не определяет случай `min == max`** (деление на ноль, сюда же `n == 1`).
   Такие входы намеренно исключены.

## Гарантии устойчивости

Тесты построены генератором со встроенными проверками, чтобы не заваливать
корректные решения на неоднозначностях:

- **Только целочисленные входные значения** — парсятся и через `%d`, и через `%lf`.
- **Нет случаев `min == max`** в Quest 1.
- **Нет пограничных округлений** в Quest 1: ни одно ожидаемое значение не попадает
  на точную середину (3-й знак ровно 5), где `%.2f` может уйти в любую сторону.
- **Quest 2 не зависит от выбора сигмы**: проверено, что вердикт совпадает и для
  популяционной (`/n`), и для выборочной (`/(n-1)`) сигмы — реализация в
  `decision.c` нам недоступна.
- **Quest 2 не стоит на заборе `3σ`**: ни одна точка не ближе 1% к границе, поэтому
  разница между `<= 3σ` и `< 3σ` роли не играет.
- **Quest 2 не стоит на константе 0.618**: ближайшее среднее — 0.62, что на два
  порядка дальше, чем расхождение между `0.618` и точным `(√5−1)/2 ≈ 0.6180339887`.

Все 27 эталонов сверены с эталонными реализациями на C (`printf("%.2f")`), а не
только с расчётом на Python.

## Классы эквивалентности

### Quest 1 — `data_module_entry/`

| Тест | Класс | Вход | Ожидание |
| ------ | ------ | ------ | ------ |
| 1 | пример из README | `5 / 1 2 3 4 5` | `0.00 0.25 0.50 0.75 1.00` |
| 2 | минимальный размер n=2 | `2 / 0 1` | `0.00 1.00` |
| 3 | n=2 через ноль | `2 / -5 5` | `0.00 1.00` |
| 4 | убывание: порядок не сортируется | `5 / 5 4 3 2 1` | `1.00 0.75 0.50 0.25 0.00` |
| 5 | неравные интервалы (min-max vs ранговая) | `3 / 1 2 4` | `0.00 0.33 1.00` |
| 6 | min и max внутри, не по краям | `4 / 10 -10 0 5` | `1.00 0.00 0.50 0.75` |
| 7 | дубликаты, min/max различны | `6 / 3 1 3 5 1 5` | `0.50 0.00 0.50 1.00 0.00 1.00` |
| 8 | все значения отрицательные | `4 / -40 -30 -20 -10` | `0.00 0.33 0.67 1.00` |
| 9 | большие величины | `3 / 1000000 2000000 3000000` | `0.00 0.50 1.00` |
| 10 | большой разброс, малые жмутся к 0.00 | `4 / 0 1 2 1000` | `0.00 0.00 0.00 1.00` |
| 11 | max первым, min последним | `4 / 9 5 7 1` | `1.00 0.50 0.75 0.00` |
| 12 | периодическая дробь, округление вниз | `3 / 0 1 3` | `0.00 0.33 1.00` |
| 13 | округления .17/.33/.67/.83 | `7 / 2 4 6 …14` | `0.00 0.17 … 1.00` |
| 14 | увеличенный n | `20 / 0 1 … 19` | `0.00 0.05 … 1.00` |
| 15 | всего два различных значения | `6 / 7 7 7 9 7 9` | `0.00 0.00 0.00 1.00 0.00 1.00` |

### Quest 2 — `yet_another_decision_module_entry/`

Решение = (правило трёх сигм выполняется) И (среднее ≥ 0.618). Классы покрывают
все четыре комбинации двух условий.

| Тест | Класс | 3σ | среднее ≥ 0.618 | Ожидание |
| ------ | ------ | ------ | ------ | ------ |
| 1 | пример из README (`1 2 3 4`) | ✓ | ✓ (2.5) | `YES` |
| 2 | минимальный размер n=2 | ✓ | ✓ (1.5) | `YES` |
| 3 | среднее 0.8 выше порога | ✓ | ✓ | `YES` |
| 4 | среднее 0.6 ниже порога | ✓ | ✗ | `NO` |
| 5 | тугая граница: среднее ровно 0.62 | ✓ | ✓ | `YES` |
| 6 | тугая граница: среднее ровно 0.60 | ✓ | ✗ | `NO` |
| 7 | среднее ровно 0 | ✓ | ✗ | `NO` |
| 8 | отрицательное среднее | ✓ | ✗ | `NO` |
| 9 | большой разброс, среднее высокое | ✓ | ✓ (30) | `YES` |
| 10 | выброс ломает 3σ, среднее высокое | ✗ | ✓ | `NO` |
| 11 | выброс ломает 3σ, среднее низкое | ✗ | ✗ | `NO` |
| 12 | среднее ровно 1.0 | ✓ | ✓ | `YES` |

> Замечание про правило трёх сигм: при малых n оно математически не может
> нарушиться — максимальный z-score равен `(n-1)/√n`, что превышает 3 только при
> `n ≥ 11`. Поэтому тесты 10 и 11 используют n=20: на меньших выборках ветку
> «3σ нарушено» проверить нечем.

## Запуск

`run_tests.sh` ищет тесты в `tests/<ветка>/<имя_программы>/`, где имя — это
`basename` файла из `src/*.c`. Директории названы по entry-файлам
(`data_module_entry.c`, `yet_another_decision_module_entry.c`), но в D09T06 они
лежат вложенно (`src/data_module/…`), а скрипт просматривает только плоский
`src/*.c` и не заходит в поддиректории — то есть сам он эти тесты не подхватит.
Прогнать можно вручную:

```bash
./build/Quest_1 < tests/D09T06/data_module_entry/test5.txt
diff <(./build/Quest_1 < tests/D09T06/data_module_entry/test5.txt) \
     tests/D09T06/data_module_entry/test5.out
```
